# ウェブ広告オーディエンスセグメント設計 Skill v4.1

### 入力補完ルール

- 【3】未整理の場合 → 【1】【2】とウェブ検索から5seg構造の仮説を立てて進める
- 【6】未入力の場合 → 【1】【2】から型を推定し「〇〇型（推定）」と明示してSTEP2-0を実行。型不明ならSTEP2-0をスキップ
- BM型マッピング詳細は `references/bm-segment-axis-map.md` を参照
- 【4】の対象5segが確定したら `references/5seg-segment-logic.md` を参照し、推奨軸・失敗パターンをSTEP2-1に反映

## 【STEP2-1：セグメント洗い出し】

5つ以上洗い出す（3つ以下では比較評価で差がつかず「全部やる」結論になりやすいため）。

**軸の決定優先順位：**

```
第1優先：5segロジックの主軸（references/5seg-segment-logic.md の該当セグ参照）
第2優先：BM型の推奨軸（STEP2-0参照）
第3優先：広告ターゲティング実現性（管理画面で定義できるか）
```

**BM型がない場合は以下5軸から「消費行動が最も変わる」軸を選ぶ：**

| セグメント軸 | 内容 | 媒体での設定例 |
|:------------|:-----|:-------------|
| デモグラフィック | 年齢・性別・世帯収入・子供の有無 | Google：詳しいユーザー属性 / Meta：詳細ターゲット |
| 購買行動 | EC利用傾向・購買意向・購入頻度 | Google：購買意向セグメント / 各媒体：カスタマーマッチ |
| 検討行動 | 検索KW・サイト閲覧・比較行動 | Google：カスタムセグメント / Yahoo!：サーチターゲティング |
| 興味関心 | 趣味嗜好・ライフスタイル | Google：アフィニティ / Meta：興味関心 |
| ライフイベント | 結婚・出産・購買サイクル位置 | Google/Meta：ライフイベント |

出力テーブル：

```
| # | セグメント名 | ペルソナ像 | 採用軸 | 広告ターゲティングでの定義方法 |
|---|------------|----------|:------|:--------------------------|
```

## 【STEP2-2：セグメント評価】

| 評価軸 | ◎ | ○ | △ |
|--------|---|---|---|
| **収益性** | LTVが高い（高単価×高リピート） | 単価かリピートの一方が高い | LTVが低い |
| **規模** | 市場規模大・検索ボリューム潤沢 | 中程度 | 母数が小さい |
| **成長性** | 検索トレンド右肩上がり | 安定横ばい | 縮小傾向 |

自社DBがない場合はウェブ検索で業界LTV傾向・類似BM型事例を調査して推定する。

```
| セグメント | 収益性 | 規模 | 成長性 | 根拠 |
|:----------|:------|:-----|:------|:-----|
```

## 【STEP2-3：総合評価と優先度決定】

**広告ターゲティング実現性：** ◎複数媒体で定義可 / ○一部媒体で粗い / △1媒体のみ

```
■ 総合評価
| セグメント | 収益性 | 規模 | 成長性 | 広告実現性 | 総合優先度 |
|:----------|:------|:-----|:------|:---------|:---------|

■ 優先度マトリクス
|  | 収益性＋広告実現性：高 | 収益性＋広告実現性：中 | 収益性＋広告実現性：低 |
|:---|:---|:---|:---|
| 規模＋成長性：高 | 【最優先】 | 【将来】 | ─ |
| 規模＋成長性：中 | 【高優先】 | 【ニッチ/維持】 | 【見直し】 |

■ 優先度判定ロジック（各セグメント3〜5行。数値根拠と戦略的判断を含める）
```

## 【STEP2-4：原因仮説とクリエイティブアングル】

最優先・高優先セグメントに対して、「なぜまだ買っていないか/なぜ離反したか」の仮説を1〜3個立てる。仮説はCTR/CVRで検証可能な形で記述し、棄却された場合の修正サイクルを前提とする。

```
| 対象セグメント | 未購入/離反理由（仮説） | 仮説の根拠 | クリエイティブアングル | 主戦場媒体 |
|:-------------|:-------------------|:----------|:--------------------|:---------|

■ 仮説 × 媒体の対応マップ
| 仮説 | Google | Meta | LINE | Yahoo! |
|:-----|:------|:-----|:-----|:------|
```

## 【STEP2-5：後続設計への申し送り】

```
■ セグメント × 仮説 × アングル × 媒体 × 予算の対応表
| 優先度 | セグメント | 採用軸 | 仮説 | アングル | 主戦場媒体 | 予算配分目安 |
|:-------|:----------|:------|:-----|:--------|:---------|:----------|

■ NGワード・トーン（全媒体共通）
BM型禁忌を自動反映。例：D-1型→「値引き・セール」禁止、A-1型→「ポイント失効系の急かし」禁止

■ 獲得後のLTV/定着率の留意点
BM型のチャーン基準をもとに「どの時点で離脱リスクが高いか」を記述
```

## 【STEP3：最終成果物① エグゼクティブ向けセグメントブリーフ】

**ファイル名：** `executive-segment-brief-[クライアント名]-[対象5seg].md`

STEP2-5まで完了したら必ずこのSTEPを実行する。**初見の意思決定者が上から読むだけで全体像を把握できる**構成にする。分析メモの貼り付けは禁止。`references/client-summary-template.md` の記述要素を参考にしつつ、以下の①〜⑥構成で1ファイルに統合する。

### 必須セクション構成（①〜⑥の順に出力する）

**冒頭：「このドキュメントの読み方」ブロック**
- ①〜⑥の各セクションが何を伝えるかを3〜5行のコードブロックで示す
- 読者が迷わず目的のセクションに飛べるようにする

**① 現状と課題**
- 「一言で言うと」ブロック：現状の構造問題を引用符付きで1〜2文に凝縮する
- KPI推移表：売上・顧客数・購入回数・LTV成長率など**定量データを必ず含める**
- 顧客構造の歪みをASCIIアート（砂時計型・台形型など）で視覚化する
- 3つの優先課題を表形式（課題 / 現状値 / 改善目標 / 改善インパクト試算）で整理する

**② 設計の核心**
- 今回の設計で最も重要な発見を罫線囲みのコードブロックで強調する（**定量データ必須**）
- As-Is → To-Be のブランド認知の変化をコードブロックで図示する

**③ セグメント一覧**
- 予算配分をASCIIバーチャートで可視化する（例：`████████████ 30%`）
- 全セグメントを1枚の表（ID / 名称 / 対象層 / 規模 / 優先度 / 予算 / 主戦場 / 一言説明）で俯瞰する

**④ 各セグメントの詳細**
- セグメントごとに「この人たちは誰か」→「なぜこの優先度か」→「広告の打ち方（✅やること / ❌やってはいけないこと）」の順で記述する
- 広告対象外セグメントには必ず代替手段（SEO・MA等）を記述する

**⑤ 施策原則・NGワード**
- NG番号付きの表形式（やってはいけないこと / 理由）で出力する
- 「禁止」だけでなく「なぜNGか」を必ず添える

**⑥ 次のアクション**
- 「今週中」「来週中」「配信開始から〇週間後」の時系列でコードブロックに整理する
- 各アクションは「DBリストを抽出する」等の**具体動詞**で書く。「検討する」は禁止
- 末尾にExcelの参照先（シート名）を案内する

### 品質チェック

- [ ] 冒頭に「このドキュメントの読み方」があるか
- [ ] ①に定量データ（KPI表）があるか
- [ ] ②の最大発見が罫線囲みコードブロックで強調されているか
- [ ] ③にASCIIバーチャートと一覧表があるか
- [ ] ④で各セグメントに「✅やること / ❌やってはいけないこと」があるか
- [ ] ⑤のNGワードに「なぜNGか」の理由があるか
- [ ] ⑥のアクションが時系列コードブロックで具体動詞になっているか

## 【STEP4：最終成果物② 広告配信設定Excel】

**ファイル名：** `ad-delivery-settings-[クライアント名]-[対象5seg].xlsx`

STEP3のブリーフで合意した前提・優先度を、**広告運用担当者がそのまま入稿に使える**形でExcelに落とす。**Markdown版オーディエンス設定書を別ファイルとして出力しない。**

生成方法：`openpyxl` / `pandas.ExcelWriter` 等で `.xlsx` を生成する。生成不可の場合はシート単位のCSVに分割する。

### Excelブック構成（シート順）

| シート名 | 内容 | 役割 |
|---------|------|------|
| `セグメント一覧` | 全セグメントの概要を1行1セグメントで整理 | 全体俯瞰・意思決定用 |
| `CRMリスト定義` | 各セグメントに対応するCRMリストの抽出条件・用途を定義 | データ担当者への依頼書 |
| `Google配信設定` | Google広告のオーディエンス・キャンペーン設定（セグメント別） | Google広告入稿用 |
| `Meta配信設定` | Meta広告のオーディエンス・キャンペーン設定（セグメント別） | Meta広告入稿用 |
| `媒体別注意事項` | Google/Metaの機能差・制約・クライアント固有のNG事項 | 運用開始前の確認用 |

### シート「セグメント一覧」の列定義

Google/Metaの詳細設定はここには含めない。セグメントの定義・優先度・方針のみを記載する。

| 列グループ | 推奨ヘッダー |
|-----------|------------|
| 識別 | セグメントID・セグメント名・対象層（5seg）・人数規模 |
| 優先度 | 優先度・予算% |
| 方針 | ペルソナ要約・課題仮説（離脱/未購入理由）・クリエイティブアングル・配信タイミング・主戦場媒体 |
| CRM | CRMリスト参照ID（CRMリスト定義シートとの対応）・インパクト試算 |

### シート「CRMリスト定義」の列定義

各セグメントで参照するCRMリスト（L1・L2・L3…）の抽出条件を定義する。Google/Meta配信設定シートのリスト名と1:1で一致させること。データ担当者への依頼書として使える粒度で記述する。

| 列 | 内容 |
|---|------|
| リストID | L1・L2・L3…（優先度カラーで色分け） |
| リスト名 | わかりやすい名称（例：「離脱_繁忙期のみ」） |
| 使用セグメント | 対応するセグメントIDと名称 |
| 推定件数 | 規模感の目安 |
| 抽出条件（SQLライク） | WHERE句形式でコメント付きで記述。データ担当者がそのまま使える粒度 |
| 使用するCRMフィールド | 必要なカラム名を列挙（メール・電話・購入日・回数等） |
| 更新頻度 | 月次 / 週次 等 |
| 最小件数 | 媒体の有効件数基準（Google/Meta共通：最小100件・推奨1,000件以上） |
| アップロード先・用途 | Google広告：カスタマーマッチ / Meta広告：顧客リストCA等 |

**フッターに必ず記載する注意事項：**
- Google広告：CSV（UTF-8）・英語ヘッダー・SHA256ハッシュ化必須
- Meta広告：CSV・SHA256ハッシュ化必須
- 共通：メール小文字化・電話はE.164形式（+81XXXXXXXXXX）

### シート「Google配信設定」の列定義

`references/google-ads-audience-settings.md` を参照して各行を埋める。1行＝1セグメント。

| 列グループ | 推奨ヘッダー |
|-----------|------------|
| 識別 | セグメントID・セグメント名・優先度 |
| オーディエンス | G_オーディエンス名・G_キャンペーンタイプ |
| 自社データ | G_自社データ（カスタマーマッチ）・G_類似/最適化 |
| デモ・地域 | G_年齢・G_性別・G_地域 |
| ターゲティング | G_アフィニティ/購買意向（1列に統合可）・G_カスタムKW・競合URL |
| 除外・配信 | G_除外設定・G_配信タイミング・G_入札モード |

**フッターに必ず記載する注意事項：**
- カスタムセグメント（KW指定）は検索・ショッピングでは使用不可
- P-MAXシグナルはハードターゲティングではなくGoogleが自動拡張する
- 類似セグメントはデマンドジェネレーション・P-MAXのみで利用可能

### シート「Meta配信設定」の列定義

`references/meta-ads-audience-settings.md` を参照して各行を埋める。1行＝1セグメント。

| 列グループ | 推奨ヘッダー |
|-----------|------------|
| 識別 | セグメントID・セグメント名・優先度 |
| オーディエンス | M_オーディエンス名・M_キャンペーンタイプ |
| 自社データ | M_自社データ（顧客リストCA）・M_類似% |
